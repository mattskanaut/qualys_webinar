Thanks Dave for the introduction and welcome everyone to our first episode in our cloud security webinar series.  Great to have you all here.

Much of the talk around changes in security practice when moving to the cloud has more to do with changes in our technology landscape and approaches to security in general, than with our deployment methods.

So perhaps a better way to think of this is: what does our current best thinking look like when talking cyber security today?

Personally, I direct anyone interested in studying this topic to read the mandates and standards, most of them are suprisingly readable and accessible. The NIST CSF, Cyber Security Framework, is a great place to start. I know, hardly bedtime reading, but it lays out very clearly an approach to security that serves us equally well in all domains, which is the purpose of it. To define security principles that can applied, regardless of the technology that's deployed.

So, when talking cloud security what has driven change isn't so much the move to cloud platforms, but rather updated thinking about security in general.

From human-scale administration to programmatic, automated security controls. Change is not unique to security, the same concepts drive other modern practices like CI/CD.

So for our session today, we thought it was important to study some real world breaches so we can get a better understand of why the practices and tools we're talking about are truly important to your organization. Breach Context: Large Financial Services Company, in 2019, quite a while ago, but it was major breach and a great case study a few different security principles.

Scale: 100 million US customers, 6 million Canadian customers affected, and it was extremely sensitive personal financial data, including credit scores and balances, payment histories and PII like social security numbers, addresses etc was leaked

Impact: $700M+ in fines, congressional hearings, CEO resignation

"Here's how the attack worked: The attacker sent specially crafted requests to the companies misconfigured Web Application Firewall running on an EC2 instance, and spoofed it into making calls to AWS's EC2 metadata service instead of the legitimate backend services. The key point is, is that these metadata calls are now in the context of the EC2 instance iself, so the service was happy to hand over an EC2 IAM credential token to the compromised WAF. The attacker then used these stolen credentials to systematically access and download encrypted customer data from S3 buckets, data it could use the same IAM role to dcrypt using calls to the AWS KMS, Key Management Service, an encryption key management tool in AWS."

## Title: How Modern Best Practices Would Have Prevented It - A breakdown of the breach and practical cloud-native prevention.

### Slide text

How Modern Best Practices Would Have Prevented It - A breakdown of the breach and practical cloud-native prevention.

WAF Misconfiguration

Web application firewall allowed SSRF attacks through improper configuration.

Prevention: Continuous Monitoring (NIST SI-4) would catch WAF misconfigurations in minutes.

SSRF Attack & Metadata Service Exploit

Attacker accessed AWS EC2 metadata service to retrieve credentials.

Prevention: Protect Information at Rest (NIST SC-28) IMDSv2 enforced by default would prevent credential access.

Over-Permissioned IAM

IAM role had excessive permissions, allowing access to multiple S3 buckets.

Prevention: Automated Enforcement (NIST AC-3) would auto-remediate over-privileged roles and Least Privilege (NIST AC-6) ensures the WAF gets minimal permissions only.

106 Million Records Exposed

Small misconfigurations created a highway to sensitive customer data.

Prevention: Boundary Protection (NIST SC-7) with network segmentation would prevent lateral movement, limiting the blast radius.

This wasn't sophisticated hacking - it was a failure to apply updated security thinking.

"Capital One followed traditional best practice. Look what they had:"

WAF for perimeter security ✓

"But they missed some modern essentials:"

 No continuous monitoring of their configuration, they had no visibility of the fact their WAF wasn’t set up to prevent Server Side Request Forgery attacks by only allowing whitelisted backend URLs, or blocking uneeded IP ranges, basic App Firewall stuff

 A basic cloud posture control, enabling IMDSv2 for AWS metadata service would have prevented this attack by forcing the WAF to run an token aquisition step to start a session with the metadata service, so it was able to assume the EC2s IAM role with a simple GET request to a local IP.  I have to be fair to Capital One at this point, this breach was what put IMDS_v2 on the radar for the industry, so perhaps this part of the chain wasn’t really their fault.

 The over-permissioned IAM role (not least privilege for the job at hand) meant it could both access the S3 bucket and dcrypt the data

 I’d add in here, no lateral movement protection, why was the WAF even able to route to the S3 bucket, that should be something only an application can reach, a miconfig like point 1 really.

So what are the lessons here?

For beginners: ”Limit things to only what is necessary for their task.  This is such a brilliant example of why this principle is important.  Why could the WAF route to the metadata service or S3 bucket at all?  Did it need to be able to?

Advanced perspective: ”Similarly, limiting permissions with tighter IAM roles would have prevented the attacker being able to access and decrypt the customer data. Finally, IMDSv2 enforcement would have blocked this entirely. This is why automated security controls are so important. Its not possible to catch things like this, at scale, without them.”

Breach Context: MOVEit (2023)

"Lets talk about MOVEit in 2023 - a breach that perfectly illustrates why tool fragmentation kills security programs. 2,100+ organizations affected, 77+ million records exposed. Attackers targeted a widely used file transfer utility, exploiting a classic SQL injection vulnerability that gave them access to end user files.

Here's what makes this story interesting: The Cl0p ransomware group initially exploited this as a zero-day, but the real damage came after a CVE was published and patches were released. While Progress Software quickly issued fixes, most organizations failed to patch their MOVEit instances in time. This turned from a zero-day problem into a classic vulnerability management failure.

The attack was particularly devastating because many organizations didn't even know they had MOVEit instances running in their cloud environments, making both discovery and patching impossible. The MOVEit breach exposed a fundamental problem: organizations had multiple security tools, but couldn't answer two basic questions: 'Where are all our MOVEit instances?' and 'How critical is this vulnerability to our environment?'

Here's how the failure cascade actually happened. Asset discovery failed - inventory tools didn't track all cloud instances. Shadow IT stikes again. Vulnerability management was fragmented - patch management couldn't correlate findings across environments. Risk assessment was disconnected from reality - CVSS scores ignored actual asset criticality and exposure. Response coordination collapsed - security teams couldn't prioritize remediation across complex infrastructure.

Each tool had a piece of the puzzle, but no one could see the complete picture of exposure and risk. CNAPP integration would have changed everything: unified asset discovery across all cloud environments, correlated vulnerability data with actual business impact, and automated risk prioritization based on real-world context, not just CVSS scores.”

CNAPP Implementation: Success vs. Traps

Success Factors

Build on Existing Tool Investments

Leverage your current security stack to avoid redundancy and maximize value

Be Clear About Your Own Requirements

Measure Business Impact

Include Development Teams Early

Engage developers from the start to foster ownership and smooth adoption.

Celebrate Quick Wins

Common Pitfalls

Isolation Implementation

Deploying CNAPP as a standalone solution without regard to your broader security ecosystem.

Over-Engineering Solutions

Implementing overly complex or unnecessary features that add friction rather than value.

Skipping the Business Case

Its not security for security's sake, you need clear focus on the business outcomes your trying to achieve

Ignoring Existing Processes

Failing to adapt CNAPP to complement or improve current security and development practices.

Perfect Security Mindset

Aiming for an unrealistic, flawless security posture from day one.

### Speakers Notes - Expert

”I’d like to close Let's be honest about what makes CNAPP implementations easy or difficult. I’d say that its hard for CNAPP tools to go completely wrong, usually some benefit is gained with even problematic deployments, but you want to keep some factors in mind to get best results from your investment of time and money.

The organizations that succeed understand they're not starting from zero. You've already invested in security tools - don't throw that away. Smart CNAPP implementation builds on what's working and fills the gaps. Don’t buy expensive CNAPP features you’ll never use because they didn't take time to understand your actual requirements first.

Successful teams also get developers involved early, not as an afterthought. When developers feel ownership in the security process, adoption happens naturally. And here's the key - those early wins in efficieny win friends with Devs. Every time automation saves someone an hour, every vulnerability that gets fixed faster because you can give them clear instructions, backed with clear justifications for the priority, is building confidence you’re not going to burn their valuable feature development time on fixes that don’t achieve anything.

Some ways teams have issues with CNAPP implementation, and to be honest, these failings are really just tool or infrastructure implementation failures in general. CNAPP, like any complex tool, is not something you drop on an organization from on high as an isolated “this will fix everything” deployment. It needs to integrate into your broader security and technology ecosystem. 

Beware the complexity trap. Often clear, simple and good enough, wins out over complicated feature sets that are difficult to operationalise. This is less about tool selection and more about what features of your CNAPP will you deploy first, not everything needs to happen day 1. 

The worst mistake? Treating this as purely a techology project without a clear business case. Your CFO doesn't care about theoretical security improvements - they care about measurable business outcomes. Make sure this is foremost in your mind, what business outcome am I achieving for this spend?

Final take away and I can’t stress this enough - the “perfect security mindset” is a problem in itself. Don’t think CNAPP will magically solve every security problem from day one. That's not how this works. CNAPP is about building better security operations, not achieving security perfection, you’ll never fix it all. Start with your most critical business risks, and expand from there."
